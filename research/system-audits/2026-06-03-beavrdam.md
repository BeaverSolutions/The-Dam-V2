# SYSTEM AUDIT - 2026-06-03 - BeavrDam

Scope: post-incident no-burn E2E code audit after the 5-lead V2.1 Research proof returned zero output. No paid provider calls and no production writes were run during this audit.

## Critical Findings Fixed

1. Research proof had no mandatory query-plan preflight.
   - Evidence: `/api/autonomous/v2-1/research-proof` previously ran `runSignalHunt()` immediately after loading ICP.
   - Fix: added `previewSignalHuntPlan()` and changed the proof endpoint into a two-step gate. `dry_run=true` returns the executable first-N query plan and hash. Actual execution now requires `confirm_query_plan_hash` matching `query_plan.query_set_hash`.
   - Files: `server/services/signalHunt.js`, `server/routes/autonomous.js`, `server/tests/unit/v2-1-research-proof-endpoint.test.mjs`, `server/tests/unit/signal-hunt-query-builder.test.mjs`.

2. Chat Signal Hunt could still run the old unsafe path.
   - Evidence: `/api/autonomous/chat` Signal Hunt branch targeted 20 leads, passed no `maxPaidQueries`, saved leads, then auto-called `directorExecute()`.
   - Fix: chat Signal Hunt now returns a preview unless `allow_paid_signal_hunt=true`; execution is capped at 5, passes `maxPaidQueries: signalLimit`, saves Research leads only, and does not auto-trigger Sales, Enforcer, approvals, or send queue.
   - Files: `server/routes/autonomous.js`, `server/tests/unit/v2-1-research-proof-endpoint.test.mjs`.

3. All-client autonomous fanout was not fail-closed when the allowlist was empty.
   - Evidence: `getEnabledClientIds()` returned every client when `AUTONOMOUS_ENABLED_CLIENTS` was unset.
   - Fix: empty allowlist now resolves to zero clients and logs the no-op. Fanout routes require explicit tenant allowlist.
   - Files: `server/routes/autonomous.js`, `server/tests/unit/p0-stabilization-contracts.test.mjs`.

## High Findings Remaining

1. RLS global pool enforcement is still off by default.
   - Evidence: `server/db/pool.js` gates tenant-aware `pool.query()` behind `RLS_ENFORCE_ENABLED === 'true'`; `server/index.js` notes it was disabled because Supavisor transaction mode did not preserve role state.
   - Risk: authenticated route safety still relies heavily on explicit `client_id` filters unless the RLS wrapper is validated and enabled.
   - Boundary: acceptable for Beaver-only proof work; not rock-solid for external tenant expansion.

2. `maxPaidQueries` is not an all-provider spend ceiling.
   - Evidence: Signal Hunt caps open-web and decision-maker search, but email enrichment can still use domain discovery, Hunter, and MillionVerifier under their own provider caps.
   - Risk: a "5 lead" proof can spend more than 5 total provider units if signals are found and enrichment proceeds.
   - Boundary: provider caps still fail closed independently; a unified per-operation provider budget remains future hardening.

## Medium Findings Remaining

1. Env examples still omit non-critical operational knobs.
   - Fixed the critical knobs: `AUTONOMOUS_ENABLED_CLIENTS`, `DB_BUILDER_ENABLED_CLIENTS`, `RLS_ENFORCE_ENABLED`, `ADMIN_SQL_ENABLED`, `DAILY_WEB_LINKEDIN_SIGNAL_CAP`, `DAILY_KICKOFF_SIGNAL_PREFILL_ENABLED`, `SIGNAL_HUNT_MAX_QUERIES`, `SIGNAL_HUNT_RESULTS_PER_QUERY`, `CHANNEL_ESCALATION_DAILY_CAP`, `MAX_DAILY_SENDS`.
   - Remaining undocumented vars are mostly scripts, model pricing, local DB fallback, optional provider aliases, or test/deploy checker knobs.

2. Local DB-backed query preview could not run from this machine.
   - Evidence: local `.env` resolves `DATABASE_URL` hostname `db`; the Docker database was not reachable.
   - Boundary: code-level preview and tests passed, but no live Supabase query-plan preview was run during this audit.
